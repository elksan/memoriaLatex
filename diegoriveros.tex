\chapter{Introducci\'{o}n}

En la presente memoria de título se definen  las principales características del Software a desarrollar: ``PRACUTAL'', el cual garantiza prestar un apoyo fundamental a la Gestión del Centro Prácticas, perteneciente a la Facultad de Ingeniería de la Universidad de Talca.

Respecto al Centro Prácticas como entidad, es necesario definir su función, la cual consiste principalmente en proveer y regular prácticas, memorias y ofertas de trabajo, a los estudiantes de la Facultad, proceso que en la actualidad resulta bastante engorroso al momento de llevar a cabo estas tareas.

Con la implementación de PRACUTAL, se resuelve una de las principales problemáticas existentes en la formación  integral y práctica de profesionales, la cual es estandarizar el proceso de toma de prácticas para todas las carreras, con el fin de optimizar y centralizar la información.

El Sistema a desarrollar surge por la necesidad de prestar apoyo a la Gestión de trabajos prácticos al interior de la Facultad de Ingeniería, con lo cual esta herramienta permitiría la administración de los trabajos de todas las carreras, recursos, tareas y demás funciones y actores relacionados con el Centro Prácticas, organismo fundamental dentro de la Universidad.

Cabe destacar que el Sistema se fundamenta en una base de requerimientos presentados por el organismo, y contempla una gran cantidad de funcionalidades, a las cuales se añaden otras concernientes netamente a aspectos internos que se desean satisfacer y cubrir dentro de la Unidad.

Con respecto a la documentación del Software, se incluyen todos los puntos principales que se deben definir en todo sistema que se desarrolla, como son: descripción general, ámbito y funciones del sistema; diseño funcional y técnico; arquitectura y tecnologías relacionadas. Todo lo anterior, sumado a la caracterización de usuarios que manejarán el sistema.

La documentación debe contener asimismo, modelos y diagramas referentes al Sistema, como son: modelos de caso de uso, con su respectiva definición y especificación; modelo de datos, diagramas de arquitectura y diseño de clases esenciales para la implementación de la aplicación.

Se incluye además, un diseño de interfaz de fácil entendimiento y cómoda utilización, siguiendo estándares globales de experiencia de usuario para un eficiente manejo de la aplicación y un máximo desempeño de las tareas de gestión y comunicación entre los actores.

\section{Descripci\'{o}n del contexto}

Esta memoria de título nace como mejora a un proyecto que consistía en la creación de una plataforma de apoyo para la labor del recientemente creado Centro Prácticas de la facultad de Ingeniería, esto durante el ramo de Taller de Software II (año 2013), el cual finalmente, no se concretó.

\section{Descripci\'{o}n del problema}
Actualmente existe la necesidad de contar con una plataforma que centralice la información de los estudiantes, organizaciones y trabajos prácticos de manera rápida y oportuna. La tarea de reunir y gestionar estos antecedentes es larga y compleja; sumando a todo esto, que el proceso es ejecutado manualmente.

\section{Propuesta soluci\'{o}n}
Para resolver esta problemática, se decide generar un software que apoye en gran manera a la gestión del Centro Prácticas, a través de un aplicativo web, en el cual se podrá realizar un flujo completo e integral del proceso actual de manera sistémica, partiendo desde que se crea una oferta de trabajo práctico; el alumno se adjudica dicha oferta; y, finalmente se evalúa su desempeño.
Dentro del flujo de trabajo, se establecen cuatro actores, que son: Organización, Centro Prácticas, Estudiante y Director de Escuela.
Este sistema será implementado de acuerdo a los detalles técnicos que a continuación se mencionan:
\begin{itemize}
	\item Lenguaje Java 1.7
	\item Ninja Framework
	\item Base de datos PostgreSQL 9.6
	\item HTML 5 con Materialize CSS
	\item Servidor Jetty (embebido en Ninja)
\end{itemize}

\section{Desarrollo de aplicaciones web}

\subsection{Cliente - Servidor}
La arquitectura cliente servidor es altamente usada en el diseño de aplicaciones web. Algunos protocolos estandarizados de comunicación cliente servidor son los siguientes:
\begin{itemize}
	\item FTP
	\item SMTP
	\item HTTP
\end{itemize}

Un sistema cliente servidor también puede ser definido como una arquitectura de software. Esta se basa en clientes y el servidor, los primeros son los realizadores de peticiones y el servidor es el encargado de responder a dichas peticiones. La arquitectura cliente servidor provee comunicación entre procesos ya que esta establece el intercambio de datos de ambas partes y cada una realiza diferentes funciones.

Beneficios de arquitectura cliente-servidor:
\begin{enumerate}
	\item Separa el procesamiento de la aplicación a través de múltiples máquinas.
	\item Reduce la replicación de datos ya que estos se almacenan en el servidor y no en cada cliente.
	\item Permite el intercambio de recursos entre los clientes y el servidor.
\end{enumerate}

\subsection{Modelo de capas}
La arquitectura de capas se compone de los siguientes elementos:
\begin{itemize}
	\item Presentación
	\item Lógica de Negocio
	\item Datos	
\end{itemize}

El objetivo de una arquitectura de N capas es intentar desacoplar estos componentes. 

La más usada es la arquitectura de 3 capas la cual separa los componentes mencionados anteriormente como capas. La ventaja de este modelo es que deja estas capas desacopladas entre sí, dando la opción de que potencialmente cada componente puede estar alojado en una máquina diferente.

Principios de esta arquitectura:
\begin{itemize}
	\item Está basado en la arquitectura cliente servidor.
	\item Cada capa debe ser independiente y no debe exponer dependencias relacionadas con la implementación.
	\item Las capas que no están conectadas no se deben comunicar entre sí.
	\item Los cambios en la plataforma solo impactan a la capa que está corriendo en la plataforma afectada.
\end{itemize}

Descripción de las capas:
\begin{itemize}
	\item Capa de presentación: \\
	Se ubica en el primer nivel la aplicación. Esta corresponde a la interfaz de usuario. Su principal función es traducir los datos en algo que el usuario pueda entender.
	\item Capa de lógica/ negocio: \\
	Su función es coordinar la aplicación, sus comandos y tomar decisiones lógicas, realizar cálculos. Además se encarga de mover los datos entre las otras dos capas.
	\item Capa de datos: \\
	Esta capa es la responsable de almacenar y consultar los datos desde una base de datos o sistema de archivos. Posteriormente, estos datos son enviados a la capa de lógica para su procesamiento y eventualmente ser presentados al usuario. 
\end{itemize}

\subsection{Modelo - Vista - Controlador (MVC)}
Modelo vista controlador se crea como solución para separar el núcleo del sistema de la presentación y la lógica de control que usa estas funcionalidades

Los componentes tienen las siguientes funcionalidades:
Modelo:
\begin{itemize}
	\item Maneja el comportamiento los datos de la aplicación.
	\item Responde a peticiones por información, usualmente de la vista.
	\item Sigue instrucciones para cambiar estados (generalmente desde el controlador).
\end{itemize}
Vista: 
\begin{itemize}
	\item Traduce el modelo a una mejor manera de interacción para el usuario. Múltiples vistas pueden estar basadas sobre el mismo modelo, para diferentes propósitos.
	\item No hace procesamiento de datos.
\end{itemize}

Controlador:
\begin{itemize}
	\item Recibe una petición del usuario y se encarga de hacer las llamadas al modelo para luego enviar una respuesta a dicha petición. Es el encargado de decirle al modelo que datos se deben obtener para presentarla al usuario.
	\item Acepta entradas desde el usuario y en virtud de esto, solicita al modelo y a la vista realizar acciones basadas en la entrada.
\end{itemize}

\subsubsection{MVC en aplicación web:}
El patrón MVC en una aplicación web se compone de las siguientes tecnologías:
Modelo:
\begin{itemize}
	\item Tablas de la base de datos
	\item Información de session
	\item Reglas de las transacciones
\end{itemize}
Vista:
\begin{itemize}
	\item HTML
	\item Hojas de estilo CSS
	\item Plantillas del lado del servidor
\end{itemize}
Controlador:
\begin{itemize}
	\item Scripts del lado del cliente
	\item Procesamientos de peticiones http
	\item Preprocesamiento de los datos
\end{itemize}

\subsubsection{Ventajas de MVC:}
\begin{itemize}
	\item Métodos del modelo ofrecen una API para los datos y estados.
	\item Facilita el diseño de la vista y el controlador.
	\item Cualquiera de los componentes puede ser reemplazado fácilmente.
	\item Muchas vistas pueden ser creadas para representar los datos de diferentes maneras.
	\item Cada vista usa la misma API del modelo.
	\item Fácil de construir y mantener.
\end{itemize}

\subsection{Web responsiva}
El diseño web responsivo o adaptativo es una técnica de diseño web que busca la correcta visualización de una misma página en distintos dispositivos, ya sean computadores de escritorio a tablets y móviles. Es por esto que hoy en día es necesario que cualquier aplicación web se adapte a los diferentes tamaños de los mismos.

\subsubsection{En qué consiste el diseño responsivo}
Se trata de redimensionar y colocar los elementos de la web de forma que se adapten al ancho de cada dispositivo permitiendo una correcta visualización y una mejor experiencia de usuario. Se caracteriza porque los layouts (contenidos) e imágenes son fluidos y se usa código media-queries de CSS3.

\begin{figure}[H]
	\vspace*{1cm}
	\includegraphics[width=\linewidth]{RESPONSIVIDAD3.png}
	\vspace*{1cm}
	\caption{Caracter\'{i}sticas del diseño responsivo}
\end{figure}

Se basa en proporcionar a todos los usuarios de una web los mismos contenidos y una experiencia de usuario lo más similar posible, frente a otras aproximaciones al desarrollo web móvil como la creación de apps, el cambio de dominio o webs servidas dinámicamente en función del dispositivo.

En definitiva, el diseño web responsiva se consolida como una de las mejores prácticas hoy en día en diseño web. Aunque como todas, tiene sus pros y contras, la web responsiva es considerada actualmente como la mejor práctica posible en el diseño web.

\chapter{Metodolog\'{i}as}

\section{Metodolog\'{i}as \'{A}giles de desarrollo}

\subsection{Agile}
Como antecedente, se tiene que los diseñadores de software durante los últimos 50 años han mantenido como premisa basar el ciclo de vida de un proyecto bajo la metodología de cascada. El asunto es que el desarrollo del proyecto nunca se lleva a cabo de manera secuencial, ya que la aparición de cambios a medida que avanza supone rectificar pasos anteriores.
Agile, se define como un marco de trabajo basado en un manifiesto publicado en el año 2001 cuyos principios son el valor del ``work in progress'', la responsabilidad de cada individuo durante el desarrollo y una actitud positiva de respuesta frente a los cambios. Corresponde a una metodología propia que acepta que los problemas no estén bien definidos y que el desarrollo del proyecto sea dinámico y esté abierto a los cambios.

\subsection{Kanban}
Kanban es un marco popular que utilizan los equipos de software que practican el desarrollo de software ágil. Esta metodología se remonta a más de 50 años.

\subsection{Tablero de Kanban}
El trabajo de todos los equipos de kanban gira en torno a un tablero de kanban, el cual corresponde a una herramienta que se utiliza para visualizar el trabajo y optimizar el flujo del trabajo entre el equipo. Aunque los tableros físicos son de gran popularidad entre algunos equipos, los tableros virtuales son una funcionalidad esencial de cualquier herramienta de desarrollo de software ágil para su seguimiento, una colaboración más sencilla y la accesibilidad desde múltiples ubicaciones.
Independientemente de si el tablero del equipo es físico o digital, su función es garantizar que el trabajo del equipo se visualice, que su flujo de trabajo se unifique y que se identifiquen y resuelvan inmediatamente todos los factores que lo bloqueen y de los que dependan. 

\subsection{Pasos de un tablero de kanban b\'{a}sico}
En un desarrollo de proyecto Agile utilizando Kanban, el proyecto se planifica dentro de un flujo de trabajo de tres pasos: 
\begin{itemize}
	\item Por hacer
	\item En curso y;
	\item Hecho. 
\end{itemize}
Cabe señalar que dependiendo del tamaño, la estructura y los objetivos del equipo, el flujo de trabajo se puede asignar para cumplir el proceso único de cualquier equipo determinado.
La metodología kanban se basa en una transparencia total del trabajo y una comunicación en tiempo real de la capacidad. Por lo tanto, el tablero de kanban debería considerarse la única fuente fiable sobre el trabajo del equipo.

\begin{figure}[H]

Descripción del Diagrama:
\begin{enumerate}

	\item El Router enruta al controlador que procesará la petición.
	\item El Controlador invoca a los servicios del negocio para procesarlo.

\end{enumerate}

Finalmente, se opta por la utilización de Ninja Framework, al ser éste un framework web Full Stack, construido con java puro, lo cual facilita la curva de aprendizaje. Adicionalmente, se consideró este framework debido a que usa tecnologías maduras (robustas) ante la experimentación.

		\item Estudiante solo puede modificar prácticas externas ingresadas por el.

\subsubsection{Tipos de usuario}

Los tipos de usuario definidos inicialmente corresponden a cuatro perfiles, los cuales se detallan a continuación:

\begin{itemize}
	\item Centro Prácticas: Rol de administrador con acceso a prácticamente a todas las funcionalidades del sistema.
	\item Director de Escuela: Su rol es validar académicamente los trabajos prácticos ofertados.
	\item Estudiante: El rol del estudiante se basa principalmente en postular a ofertas vigentes y llevar registro de su trabajo. 
	\item Organización: Tiene como función, ofertar trabajos prácticos, seleccionar y evaluar a los estudiantes.
\end{itemize}

\begin{itemize}

	\item Visualizar evaluación de trabajo práctico.
	\item Revisar si oferta de práctica cumple con los requisitos que la universidad establece.
	\item Enviar candidatos a la Organización.
	\item Mantener módulo de administración del sistema.
\end{itemize}
Director de escuela:
\begin{itemize}
	\item Ofertar prácticas para alumnos de la facultad.
	\item Visualizar evaluación de trabajo práctico.
	\item Revisar si oferta de práctica cumple con los requisitos que la universidad establece.
	\item Enviar candidatos a la Organización. 
\end{itemize}
Estudiante:
\begin{itemize}
	\item Ingresar prácticas conseguidas de manera individual.
	\item Postular a alguna práctica ofertada en la plataforma.
	\item Llevar una bitácora de la práctica adjudicada.
\end{itemize}
Organización:
\begin{itemize}
	\item Ofertar prácticas para alumnos de la facultad según sus habilidades y conocimientos.
	\item Escoger potenciales candidatos para adjudicarse una práctica.
	\item Evaluar trabajo práctico del alumno una vez concluido.
\end{itemize}

\section{Etapa exploraci\'{o}n de aplicaci\'{o}n web}
Esta etapa corresponde al análisis de algunas de las tecnologías que son utilizadas hoy en día en el desarrollo de aplicaciones web.

\subsection{Bootstrap, Foundation, Materialize}
Dentro del análisis de prefactibilidad del desarrollo de la aplicación, se tiene como objetivo poder afianzar y fortalecer mis capacidades. Dado lo anterior, se establecen una serie de consideraciones necesarias para la implementación del software de manera eficaz y eficiente, las que se indican a continuación:

\begin{itemize}
	\item Aplicación con interfaz llamativa y amigable para el usuario.
	\item Lograr integración rápida entre lógica de negocio e interfaz gráfica.
	\item Optimizar tiempo de diseño gráfico.
	\item Aplicación responsiva.
\end{itemize}

Con estos puntos como base, es que se analizan las tendencias de diseño web, donde algunas de las principales son: Bootstrap, Foundation y Materialize, de los cuales podemos comparar su principal característica como front-end.

\begin{table}[h]
	\vspace*{1cm}
	\begin{tabular}{|p{4.6cm}|p{4.6cm}|p{4.6cm}|}
		\hline
		\textbf{Bootstrap} & \textbf{Foundation} & \textbf{Materialize} \\ \hline
		Es el framework basado en HTML, CSS y JS más popular en estos tiempos, sirve para desarrollar interfaces de usuario con interacción. & Framework front-end avanzado, el cual es utilizado para desarrollo de aplicaciones responsivas. & Moderno framework front-end que está basado en los principios de Material Design de Google. \\ \hline
		\end{tabular}
	\vspace*{1cm}
	\caption{Cuadro descriptivo de opciones frameworks frontend}
\end{table}

De acuerdo a lo investigado y a la experiencia previa, se decide utilizar Materialize, debido a que sus componentes por defecto son atractivos y no necesita demasiada intervención para que su interfaz visual sea amigable y de bonito aspecto. 
Por el contrario, se descarta Bootstrap, ya que su estándar de diseño está muy repetido en aplicaciones web y se requiere de demasiado conocimiento o inversión en aprendizaje para lograr una interfaz visualmente llamativa. Asimismo, no se opta por Foundation ya que debía realizar un aprendizaje desde cero para lograr extraer su máximo potencial y al igual que Bootstrap se necesita tener una mayor noción en diseño web.

\subsection{Prueba de concepto (POC)}
Para comprobar si es viable la utilización de Ninja Framework, resulta necesario poder realizar una prueba de concepto, para determinar si efectivamente es factible montar una aplicación web con las características de este proyecto y permite tener una visión del esfuerzo a incurrir en el desarrollo.
Se detalla experiencia a continuación:
\begin{enumerate}
\item Se procede a instalar jdk versión 1.7 en máquina local.
\item Siguiendo la documentación del sitio de www.ninjaframework.org se descarga versión Maven actualizada de acuerdo a sitio oficial.
\item Se procede a descargar el arquetipo de ejemplo de Ninja Framework.
\item Dicho ejemplo cuenta con las operaciones básicas de cualquier aplicación (CRUD) y se prueba su correcto funcionamiento.
\item Cabe señalar, que con el ejemplo y documentación, se logra entender y aprender cómo funciona Ninja y a su vez se ajusta para el desarrollo del proyecto.
\item Dadas las pruebas exitosas, se procede a reemplazar Bootstrap por Materialize, debido a que el primero se encuentra embebido en el framework.
\end{enumerate}
